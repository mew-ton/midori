# エコシステム整備 TODO

> ステータス：計画フェーズ
> 最終更新：2026-04-23

コミュニティを外部から迎え入れる前に整備すべき、**契約・規約・導線** の TODO を列挙する。

Midori は本質的に「他人同士の協業契約を定義するプロジェクト」であり、層分離（ドライバー / アダプター種別定義 / 変換グラフ）は独立した貢献者が手を入れる前提の境界として機能する。契約の安定性が崩れると過去のプラグインが動かなくなり、作者が離れ、エコシステムが死ぬ。**コミュニティがゼロの段階こそ、契約を破壊的に変更できる最後のチャンス** である。

本ドキュメントはその準備作業を tracking するもので、完了項目は順次削除せず ✅ で残す（意思決定の履歴）。

---

## 背景：想定する立ち上げシナリオ

VRChat 内の **個人開発集会・ガジェット愛好会** をファーストターゲットとし、エンジニアを最初の貢献者として迎える。集会で「見せる → 興味 → 定着」のファネルを回す前提で、「定着」を妨げる導線上の障害を潰すことが本 TODO の目的。

```
LT / 雑談で興味
  ↓
GitHub / Discord 導線
  ↓
Quickstart を 30 分で完走
  ↓
"good first plugin" から最初の貢献
  ↓
自作ハード・プラグインを持ち寄って相談
```

## フェーズ別の整備粒度

無人のプロジェクトが重厚なガバナンスを先に整備すると誰も近寄らない。規模に応じた段階設計とする。

| フェーズ | コミュニティ規模 | 要整備 |
|---|---|---|
| Phase 0（現在） | 本人のみ | ライセンス、SDK 互換ポリシー、RFC テンプレ、CoC、貢献ガイド、命名規約 |
| Phase 1 | 〜10 人 | PR 受け入れ基準、Discord 運用、Certified 試験運用 |
| Phase 2 | 〜50 人 | 意思決定 RFC 化、コミット権限の複数化、Certified 本格運用 |
| Phase 3 | 100 人〜 | 財団化検討、法人化判断、商標管理 |

Phase 0 の項目は **薄い紙 1 枚で構わない**。存在すること自体が重要。
