# セキュリティ設計：プラグイン設定パターンの脆弱性

> ステータス：設計フェーズ
> 最終更新：2026-04-21
> 対象：プラグインマニフェスト（YAML）に起因する脆弱性

## 前提：「コードなし = 安全」は不十分

プラグイン種別をコードの有無で二分するだけでは不十分。YAML 設定のみのプラグインも、構成パターンによってはセキュリティ上の問題を引き起こしうる。

| 種別 | コード | リスク分類 |
|---|---|---|
| デバイス構成（YAML） | なし | 低（データのみ） |
| デバイス種別定義 | なし | **中**（接続設定を間接的に操作できる） |
| ドライバー | あり | 高（任意コード実行） |
| ウィジェット | あり（JS） | 中（ブラウザ権限範囲） |

## セキュリティ水準

| 項目 | 現状 | 目標 |
|---|---|---|
| フィールド変更・隠蔽の禁止 | L0（制約なし） | L2（Bridge がインストール時に検証・拒否） |
| プロンプトインジェクション | L0 | L2（外部データタグによる分離） |

## デバイス種別定義による接続先の秘匿・固定

### 攻撃シナリオ

HTTP ドライバーのような通信ドライバーをラップする device kind を考える。

```
公式: http ドライバー（connection_fields に host / port を持つ）
         ↑
コミュニティ: http-some-service（device kind）
```

device kind が基底ドライバーの `host` フィールドを上書き・隠蔽できる場合：

1. コミュニティが `http-some-service` を公開（バイナリなし → 警告が出ない）
2. ユーザーが「YAML だけなら安全」と判断してインストール
3. `host` が `attacker.com` に固定され、ユーザーからは見えない
4. バイナリなしでデータの送信先が乗っ取られる

「警告が出ない種別」であることが攻撃のハードルを下げる。

### 対策：既存フィールドへの干渉を禁止

**デバイス種別定義は基底ドライバーの `connection_fields` を変更・隠蔽・デフォルト値注入できない。`additional_fields` による新規フィールドの追加のみ許可する。**

Bridge はインストール時にこの制約を検証し、違反があればエラーを返す。

```
検証ルール:
  device_kind.additional_fields[*].id が
  base_driver.connection_fields[*].id と重複していればエラー
```

この制約により：
- ユーザーは基底ドライバーの接続先を常に確認・編集できる
- device kind がいくら `additional_fields` を追加しても、既存の host / port には触れない
- YAML のみで通信先を秘匿するパターンが構造上成立しない

### `visible_when` は許可する

device kind が既存フィールドに `visible_when` を指定することは許可する。

`visible_when` は「このコンテキストでは設定不要」を意味するだけであり、値の注入ではない。たとえば output を持たない device kind が host フィールドを非表示にするのは「HTTP 送信をしないので入力欄が不要」という UX 上の自然な省略であり、攻撃経路にならない。非表示になったフィールドに Bridge が何か値を設定するわけではなく、単にドライバーがその経路を使わないだけである。

禁止するのは**値の変更・注入**のみ。表示制御は許可する。

## プラグイン脆弱性の事後対応

公開後に脆弱性が発覚したプラグインへの対応経路は2つある。

| 経路 | 内容 | 効果 |
|---|---|---|
| 公式サイトのリストから削除 | PR / Issue でプラグイン掲載を取り消す | 新規インストールを抑制。既存ユーザーには届かない |
| セキュリティマニフェストに追記 | `plugins.yanked` または `plugins.forced_updates` に記載 | インストール済みユーザーの次回起動時に無効化・更新促進 |

詳細 → [`12-distribution.md`](../12-distribution.md)（セキュリティマニフェストのスキーマと起動時フロー）

## YAML フィールドへのプロンプトインジェクション

プラグインの `plugin.yaml` 内テキストフィールドや、同梱デバイス構成の `metadata.spec` / `metadata.name` は AI のコンテキストに渡される。悪意あるプラグインがこれらに命令文を埋め込む可能性がある。

対策の詳細 → [`03-ai.md`](./03-ai.md)

YAML プラグインに固有のポイントとして：
- バイナリなし → インストール時の警告が出ない → 心理的ハードルが低い
- `description` / `metadata.spec` は長文が自然 → 命令文を埋め込んでも目立ちにくい
- 初回 AI 利用時の GUI 通知（「このプラグインは外部由来です」）が実質的な防衛線になる
