## インストール方法

### 配布窓口

| 窓口 | 位置づけ | 対象 |
|---|---|---|
| **公式サイト** | メイン。実行ファイルを直接ホスト | すべてのユーザー |
| **Booth（¥0）** | 副次窓口。同一ファイルを掲載 | VRC コミュニティ |

Booth は同じバイナリを掲載する。リリース時は公式サイトと Booth の両方を更新する（CI による自動化を前提）。Booth の「購入者への更新メール通知」機能がコミュニティへのリーチとして機能する。

### 発見性要件

- **マニュアルサイト**：操作説明・設定リファレンスを公開する
- **SEO**：公式サイト・マニュアルを検索エンジンから発見できるようにする
- **コミュニティへの宣伝**：VRC コミュニティ向け発信（SNS・Booth 等）

App Store（Mac App Store / Microsoft Store）は初期スコープ外。

### コミュニティプラグインの発見性

公式サイトをオープンソースリポジトリとして公開し、プラグイン作者が PR または Issue を作成することでプラグインへのリンクを掲載する静的ページを設ける。

```
プラグイン作者
  → 公式サイトリポジトリに PR / Issue を作成
  → レビュー・マージ
  → 公式サイトのプラグイン一覧ページに掲載
  → ユーザーはリンクから GitHub リポジトリへ到達し、URL を Midori に貼ってインストール
```

| 観点 | 内容 |
|---|---|
| 掲載内容 | プラグイン名・作者・リポジトリ URL・説明文 |
| 管理コスト | インフラ不要。静的サイトのテキスト変更のみ |
| AI レビュー支援 | PR 受信時に CI が linked リポジトリを AI で解析しレビューコメントを自動生成。メンテナの判断を補助する。public リポジトリを起点とする設計の利点 |
| 信頼性 | 掲載 = メンテナによる最低限のレビューを経たことを意味する。セキュリティ保証ではない |
| 削除 | メンテナンス放棄・悪質なプラグインは PR / Issue で削除できる |

インストール時のセキュリティ警告・サンドボックスは掲載有無に関わらず同等に適用される（`11-security/01-driver-sandbox.md`）。ドライバーのサンドボックス要件（`11-security/01-driver-sandbox.md`）との整合が必要なため、後続フェーズで検討する。

## アップデート方法

### 基本方針：通知 ＋ 任意適用

Midori の主要な使用シーンは**配信直前・演奏直前**であり、パフォーマンス要件が高い。起動中に強制再起動を踏ませることはユーザー体験上許容できない。

| 区分 | 適用方式 | タイミング |
|---|---|---|
| 通常アップデート | 通知のみ。適用はユーザー操作 | バックグラウンドでダウンロード後、起動時または Preferences から適用 |
| クリティカルなセキュリティ修正 | 強制適用 | **次回起動時**に適用。起動中の強制再起動はしない |

「クリティカル」の判断はセキュリティマニフェストで管理する（後述）。

### セキュリティマニフェスト

公式サイトに静的 JSON ファイルとして同梱する。アプリは**起動時に取得**し、必要な対処を行う。GitHub Releases を見に行く必要がない。

エンドポイントはアプリ用とプラグイン用で分離する。

```
https://midori.example.com/security/app.json
https://midori.example.com/security/plugins/{plugin-name}.json
```

プラグインは**インストール済みのものだけ**取得する。ファイルが存在しない（404）場合は問題なしとみなす。全プラグイン分をまとめて取得する必要がないため、プラグイン数が増えてもファイルが肥大化しない。

#### app.json スキーマ

```json
{
  "forced_below": "1.2.3",
  "forced_reason": "critical RCE vulnerability",
  "latest": "1.5.0"
}
```

#### plugins/{plugin-name}.json スキーマ

```json
{
  "listed": true,
  "yanked": ["1.0.0", "1.1.0"],
  "forced_below": "1.2.0",
  "reason": "malicious code detected in driver"
}
```

| フィールド | 説明 |
|---|---|
| `listed` | `true` = 公式サイト掲載済み。`false` = 掲載取り消し済み。省略不可 |
| `yanked` | 無効化対象のバージョンリスト。省略可 |
| `forced_below` | それ未満のバージョンに強制更新を促す。省略可 |
| `reason` | ユーザーに提示する理由。省略可 |

ファイルの存在自体が「このプラグインは公式サイトに認知されている」ことを意味する。**404 = 公式サイト未掲載・完全に未知のプラグイン**。

| 状態 | ファイル | listed |
|---|---|---|
| 掲載済み・問題なし | あり | `true` |
| 掲載済み・セキュリティ問題あり | あり | `true` |
| 掲載取り消し・セキュリティ問題あり | あり | `false` |
| 完全に未知 | なし（404） | — |

GUI はインストール済みプラグインに「公式サイト未掲載」バッジを表示できる。

#### プラグイン提供のセキュリティマニフェスト

プラグイン開発者も自分のリポジトリ内に `.midori-plugin/security.json` を置くことで、独自に yanked バージョンや強制アップデートを宣言できる。

```json
{
  "yanked": ["1.0.0"],
  "forced_below": "1.1.0",
  "reason": "critical bug in MIDI handling"
}
```

`listed` フィールドは持たない（公式マニフェスト専用）。

Midori は起動時にプラグインの git remote から raw URL を構築して fetch する。ディスク上のクローン済みファイルは古い可能性があるため、ネットワーク取得を優先し、失敗時にディスク上のものを使う。

#### マニフェストのマージ

同一プラグインについて公式マニフェストとプラグイン提供マニフェストの両方が存在する場合、**より厳しい方を適用**する。公式マニフェストはプラグイン提供マニフェストを上書き・強化できる。

| フィールド | マージルール |
|---|---|
| `yanked` | 両方のリストの和集合 |
| `forced_below` | より高いバージョン（制限が強い方）を採用 |

#### 起動時の処理フロー

```
アプリ起動
  → security/app.json を取得
      失敗時はキャッシュを使用。キャッシュもなければスキップ
  ↓
app.forced_below チェック
  → 現バージョンが forced_below 未満
      → 強制アップデート画面を表示し他の操作をブロック
  → それ以外
      → 通常起動。latest と差があれば通知バナーを表示
  ↓
インストール済みプラグインごとに以下を並行取得してマージ
    ① security/plugins/{name}.json（公式マニフェスト）
        404 → 問題なし
    ② .midori-plugin/security.json（プラグイン提供マニフェスト）
        失敗 → スキップ
    → yanked・forced_below はより厳しい方を採用
  ↓
  → yanked に現在のバージョンが含まれる
      → そのプラグインを無効化して起動を継続。警告を表示
  → forced_below 未満のバージョンを使用中
      → 更新を促す通知。更新まで該当プラグインを無効化
```

起動中の強制再起動はしない。アプリの強制アップデートも「次回起動時にブロック」であり、現在のセッションは継続できる。

#### フェイルセーフ

マニフェスト取得失敗時（オフライン・サイトダウン等）はキャッシュした最終取得値を使う。キャッシュもない場合は取得失敗をログに残し、強制アップデートチェックをスキップして起動を継続する。

### 実装

`electron-updater` ベース。更新ダウンロードは起動時にバックグラウンドで行う。セキュリティマニフェストの取得は `electron-updater` とは独立した fetch で行う。

## アップデート情報の提示

更新適用後の初回起動時に、ハイライト（主要な変更点）をモーダルまたはバナーで表示する。全 changelog は GitHub Releases ページへのリンクで案内する。

| 要素 | 内容 |
|---|---|
| ハイライト | 新機能・重要な変更点を箇条書き（3〜5 件程度） |
| バージョン番号 | 適用されたバージョンを明示 |
| リンク | GitHub Releases ページへの「詳細を見る」リンク |

## 利用規約

> ステータス：内容は後回し。枠組みのみ確定。

### 同意フロー

| タイミング | 内容 |
|---|---|
| 初回起動時 | 利用規約への同意を必須とする |
| 重要な変更時 | 変更内容を提示し、再同意を必須とする |

### 利用規約のアップデート

重要でない変更（誤字修正・文言整理等）は通知のみ。条件や責任範囲に関わる重要な変更は再同意フローを経る。何が「重要な変更」に当たるかはリリース時にメンテナが判断する。
