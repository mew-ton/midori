# AI 機能設計

## 概要

Midori は初期実装から AI アシスタント機能を組み込む。非エンジニアがデバイス構成・マッパーを自然言語で作成できることを主目的とし、画面操作・設定の編集・保存までを AI に委譲できる。

---

## AI プロバイダー

ローカル AI とリモート AI を切り替えて利用できる。

| 種別 | 実装 | 備考 |
|---|---|---|
| ローカル | Ollama（OpenAI 互換 API） | オフライン利用可。モデルはユーザーが管理 |
| リモート | Claude API（Anthropic） | 高精度。API キー要 |
| リモート | OpenAI API | 互換性重視の代替 |

接続設定は `preferences.yaml` で管理する（API キーは keychain / env 経由）。プロバイダーはいつでも切り替え可能。

```yaml
# preferences.yaml（ai セクション）
ai:
  provider: claude          # claude | openai | ollama
  model: claude-opus-4-6   # 省略時はプロバイダーのデフォルト
  ollama:
    base_url: http://localhost:11434
  claude:
    api_key_env: ANTHROPIC_API_KEY
  openai:
    api_key_env: OPENAI_API_KEY
    base_url: https://api.openai.com/v1   # 省略可
```

## エージェント設計

AI はツール呼び出しが可能なエージェントとして動作する。会話を通じてアプリの状態を読み書きし、画面遷移・ファイル保存まで実行できる。

### エージェントが持つツール

| ツール | 内容 |
|---|---|
| `get_app_state` | 現在開いている画面・ファイルパス・ブリッジ状態を取得 |
| `navigate` | 画面遷移（ダッシュボード / デバイス構成編集 / 変換グラフ編集 / プロファイル詳細） |
| `list_files` | devices / mappers / profiles ディレクトリのファイル一覧取得 |
| `read_file` | 指定した YAML ファイルの内容取得 |
| `read_spec_document` | 仕様書 URL（公式マニュアル等）を取得してテキストとして読み込む |
| `write_file` | デバイス構成 / 変換グラフ / プロファイルの YAML を書き込み・保存 |
| `patch_file` | 既存 YAML の一部を更新（フィールド単位の差分適用） |
| `sample_raw_events` | 接続中デバイスから raw events を一定時間サンプリング |
| `get_vrchat_params` | アバターパラメーター JSON を読み込んで一覧取得 |
| `start_bridge` / `stop_bridge` | ブリッジの起動・停止 |

### AI コンテキストの構成レイヤー

AI がマッパー生成等を行う際のコンテキストは、以下の4レイヤーで構成される。

| レイヤー | 内容 | 管理主体 |
|---|---|---|
| **Skills** | Midori 設定ファイルの記述ルール（デバイス構成・マッパー・プロファイルの YAML 構文） | アプリのビルトイン |
| **ドライバー知識** | プロトコル仕様（MIDI チャンネルメッセージの意味・OSC アドレス形式・VRChat パラメーター型の値域等） | アプリのビルトイン |
| **デバイス構成 spec** | 物理構造と操作（入力側）/ パラメーターの視覚効果と設計意図（出力側） | `metadata.spec`（ファイルに埋め込み） |
| **会話コンテキスト** | ユーザーが口頭で補足した意図・制約 | チャット履歴 |

上位2レイヤーはアプリが標準で保持する。`spec` にはデバイス固有の情報だけを書く。この分離によりトークン消費を最小に抑えつつ、AI が正確な設定ファイルを生成するのに必要な文脈が揃う。

### Skills の内容

各 Skill は Markdown ファイルとしてアプリにバンドルされる。運用方法は Claude Agent Skills と同じ——操作対象に応じて必要な Skill だけをコンテキストにロードし、AI に渡す。

各 Skill は AI がそのファイルを**読み書きできる**レベルの知識を提供する。操作手順ではなく、構文・制約・典型パターンを記述する。

| Skill | 内容 |
|---|---|
| `device-config` | `direction` / `definition` / `binding` / `layout` の構文。component type 一覧・`set` / `setMap` / `set.expr` の使い分け・`mirror` の制約 |
| `mapper` | ノードグラフの構造・ノード種別・ポート型・Array 型と scalar 型の接続ルール |
| `profile` | 入力デバイス構成・マッパー・出力デバイス構成の束ね方・driver ごとの接続設定 |
| `driver/midi` | MIDI binding の `from` フィールド（noteOn / CC / SysEx 等）と値域 |
| `driver/osc` | OSC binding の `from` / `to` フィールド |
| `driver/osc-vrchat` | VRChat 固有の制約（パラメーター型・bit 予算・アドレス形式） |

マッパー生成時は `mapper` + 関連 `driver/*` のみ、デバイス構成生成時は `device-config` + 対象 `driver/*` のみをロードする。

### 典型的な利用フロー A：仕様書からデバイス構成を生成する

ユーザーが公式 MIDI リファレンスの URL を渡す場合のフロー。

```
ユーザー: 「この URL の仕様書をもとに ELS-03 のデバイス構成を作ってください」
          https://jp.yamaha.com/files/.../ELS-03_midi.pdf

AI:
  1. [read_spec_document] → URL からドキュメントを取得して全文読み込み
  2. チャンネル割り当て・SysEx・CC・範囲を解析して YAML の binding / definition を構築
  3. 解析内容を metadata.spec に要約して書き込み（spec_source に渡された URL を記録）
  4. [write_file] → devices/yamaha-els03.yaml を保存
  5. 「保存しました。次にマッパーを作りますか？」と提示
```

`spec_source` は URL のみ受け付ける。ローカルファイルはデバイス構成の配布時に参照が壊れるため不可。公式公開ドキュメントが存在しない場合は `spec_source` を省略し、ユーザーが仕様の要点を口頭で伝えたうえで AI が `spec` に書き起こす。

`metadata.spec` に仕様要約が埋め込まれるため、次回以降は仕様書なしで AI がこのデバイスを扱える。

### 典型的な利用フロー B：既存デバイス構成をもとにマッパーを生成する

```
ユーザー: 「ヤマハのエレクトーンを接続したので鍵盤とペダルを VRChat に同期したい」

AI:
  1. [get_app_state] → ダッシュボード表示中を確認
  2. [list_files] → devices/ に yamaha-els03.yaml を発見
  3. [read_file] → metadata.spec を読み込んで楽器仕様をコンテキストに積む
  4. [sample_raw_events] → 実機から raw events をサンプリングして仕様を補完
  5. [get_vrchat_params] → アバターパラメーター一覧を取得
  6. 変換グラフ YAML 案を生成
  7. デバイス構成（出力）の YAML 案を生成
  8. [write_file] × 2 → ファイルを保存
  9. [navigate] → プロファイル詳細画面へ遷移
  10. ユーザーに接続確認を促す
```

## UI / チャットパネル

### 配置

全画面共通の **フローティングチャットパネル** として画面右端に配置する。どの画面からでもアクセスできる。

```
┌─────────────────────────────────────────┬──────────────┐
│                                         │ AI チャット   │
│  （メインコンテンツ）                     │              │
│                                         │  ・・・       │
│                                         │              │
│                                         │ [入力欄]      │
└─────────────────────────────────────────┴──────────────┘
```

パネルは折りたたみ可能。折りたたみ時は右端にアイコンのみ表示する。

```
┌─────────────────────────────────────────┐ ┌─┐
│                                         │ │🤖│  ← 展開ボタン
│  （メインコンテンツ）                     │ │  │
│                                         │ └─┘
└─────────────────────────────────────────┘
```

### チャットパネルの要素

| 要素 | 内容 |
|---|---|
| メッセージ履歴 | ユーザーとAIの会話。AI のツール実行はインラインに折りたたみ表示 |
| ツール実行インジケーター | AI がツールを呼び出す際にアイコン+名前を表示（例: `📂 ファイルを保存中…`） |
| 適用済みバッジ | `write_file` / `patch_file` 実行後のメッセージに `✓ 保存済み` バッジを表示 |
| 入力欄 | テキスト入力。`Enter` で送信、`Shift+Enter` で改行 |
| 停止ボタン | AI の応答生成中に表示。クリックでストリームを中断 |
| コンテキスト表示 | 現在開いている画面・ファイルを小さく表示（AI が参照している状態を可視化） |

### ツール実行の表示

```
AI:  鍵盤の MIDI イベントをサンプリングします。
     ▶ raw_events のサンプリング中… (3秒)
     ✓ 32 件取得

     デバイス構成を作成しました。
     ▶ devices/els03.yaml を保存中…
     ✓ 保存済み

     [プロファイル詳細画面を開く] ← アクションボタン
```

AI が生成したファイルはメッセージ内に diff ビューで折りたたみ表示できる（「内容を確認する」トグル）。

## セキュリティ

| 項目 | 方針 |
|---|---|
| ファイルアクセス範囲 | `write_file` / `patch_file` は preferences で許可されたパス（devices / mappers / profiles）のみ |
| ネットワーク | AI プロバイダーへのリクエストはメインプロセス経由のみ。レンダラーから直接 API キーに触れない |
| ブリッジ操作 | `start_bridge` / `stop_bridge` は確認ダイアログを経由する（自動実行は不可） |
| API キー | keychain または環境変数から取得。`preferences.yaml` には保存しない |

## 設定画面への追加

`preferences.yaml` の編集 UI（既存の設定画面）に **AI** セクションを追加する。

| 設定項目 | 内容 |
|---|---|
| プロバイダー | `claude` / `openai` / `ollama` をラジオボタンで選択 |
| モデル | 選択中プロバイダーのモデル名（テキスト入力またはドロップダウン） |
| API キー | 入力欄（`●●●●` でマスク。保存先は keychain / env） |
| Ollama URL | Ollama 選択時のみ表示 |
| 接続テスト | 設定した接続先へ疎通確認。OK / NG をインラインで表示 |
