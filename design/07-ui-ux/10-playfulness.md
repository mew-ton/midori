# 遊び心

UI から **人の判断の痕跡** が見える状態を指す。マスコット・キャラクター・擬人化のことではない。

AI によるデザイン生成は「安全な平均」に収束しやすく、結果として誰が作ったか分からない UI になる。Midori は量産 UI に流れないための配分ルールを持つ。

---

## 遊び心は層に宿る

UI を 3 層に分け、**どの層で遊んでよいか**を明示する。機能契約層はメタファー駆動（[`01-principles.md`](01-principles.md)）に従って手堅く作り、残り 2 層に自由度を持たせる。

| 層 | 性格 | 遊びの余地 |
|---|---|---|
| **機能契約層** | 「この UI は何を約束するか」。メタファー駆動で既知性を優先し、学習可能性を担保する | なし |
| **装飾層** | 機能契約の上に乗る視覚表現（色・曲線・減衰・静止時の挙動・効果音） | あり。設計者とユーザーが振れる |
| **フロー外** | ユーザーがタスクを遂行していない瞬間（空状態・初回起動・リリースノート・About・チュートリアル・ログ文面） | 最大。設計者が堂々と遊んでよい |

### 判定基準

**その箇所で遊んだ結果、ユーザーのタスク完了が損なわれるか？**

- 損なわれる → 機能契約層。遊ばない
- 損なわれない → 装飾層 or フロー外。遊んでよい

---

## 機能契約層（遊ばない）

メタファーが「この UI は何を約束するか」を決める。契約は崩さない。

| メタファー | 契約 | 崩してはいけない点 |
|---|---|---|
| オシロスコープ（プレビュー計器） | 信号レベルが時系列で読める | 縦軸 = 値・横軸 = 時間の関係／読み取れる静止状態 |
| IC チップ（デバイスブロック） | 両側にピン（ポート）が並ぶ | ポート位置・型識別子（形状＋色）／ラベル可読性 |
| ジャンパーワイヤー（接続線） | 端子から端子へ物理的につながっている状態 | 接続／未接続の区別／型色と形状の二重エンコード |
| パッチパネル（バインディング） | 端子と端子の対応付け | 対応関係の明示性／エラー時の赤表示 |
| ブレッドボード＋導線（全体フロー） | 左→右の信号方向 | 方向の一貫性／論理的な到達可能性の可読性 |

機能契約層では **既知性が学習可能性を担保する価値として働く**。既知の UI パターン（ナビゲーションバー・フォーム等）は、メタファー／画面の意味構造と一致する限り採用してよい（[`01-principles.md`](01-principles.md) の「各パーツに説得力を持たせる」を満たす）。「よくある UI だから」だけでは根拠にならない点は変わらないが、既知性を理由として積み上げることは可能。

## 装飾層（遊ぶ）

機能契約を崩さない範囲で、視覚表現に人の判断を入れる。

### 設計者が遊ぶ例

- ポート接続時のスナップ挙動（吸着のイージング、接続成立音の有無と音色）
- 接続線の被覆テクスチャ・描画カーブ
- オシロスコープの針の減衰カーブ、静止時の微振動
- 実行ボタン `▶` の押下フィードバック
- デバイスブロックの折りたたみアニメーション
- 起動時のタイトルバーロゴの描画タイミング

### ユーザーに開く余地

Midori 自体はニュートラルに留め、ユーザー側で温度を振れるようにする。IDE モード（装飾最小）でもホビーモード（キャラ入り）でも居場所がある、というスタンス。

アプリが用意する「遊びの蓋」：

| 項目 | ニュートラル側 | 振れる側 |
|---|---|---|
| インジケーター表現 | オシロスコープ | 棒グラフ／キャラクター絵／カスタム画像 |
| デバイスアクセントカラー | 自動割り当ての muted 8 色 | ユーザーが選ぶ |
| 効果音（接続成立・実行開始・エラー） | OFF がデフォルト | ユーザーが音色選択 |
| 背景 | モノクロサーフェス | 壁紙・パターン |
| 起動画面 | ロゴのみ | ユーザー設定画像 |

**契約（読み取れる情報）は差し替え後も保存される。** オシロをキャラ絵に差し替えても「信号レベルが読める」ことは維持する（例：キャラがジャンプする高さが値に対応する）。契約を破壊する差し替えは提供しない。

## フロー外領域（最大の自由）

ユーザーがタスク中でない瞬間は、設計者が**特定の言葉・特定の書きぶり**を選ぶ場所。ここで思考停止の既存テンプレを使うと一気に量産 UI 化する。

適用箇所：

- **空状態のコピー**（プロファイルが 0 個・変換グラフが未作成 等）
- **初回起動・ワークスペース作成完了の文面**
- **リリースノート**（アップデートダイアログの変更点説明）
- **チュートリアル・ツールチップの語り口**
- **ログメッセージの書き方**（ブリッジ開発者視点が出る場所）
- **About 画面・クレジット・バージョン番号の表示**
- **非クリティカルな通知トースト**

フロー外でも以下は平易・具体性を優先する（[`01-principles.md`](01-principles.md)）：

- フォームラベル・設定項目名
- エラーメッセージのうち**重大なもの**（原因と対処を具体的に書くことを優先、ユーモアは副次）
- YAML キー・API 契約（[`00-naming.md`](../00-naming.md) に従う）

## アンチパターン

遊び心は 2 方向に逸れる。

### 機能契約層で奇をてらう

- オシロスコープが抽象アートになって信号が読めない
- 型色と形状の対応を独自解釈で変える
- ナビゲーション位置を毎画面変える

**既知性の放棄は遊び心ではない。学習コストの押し付け。**

### 装飾層・フロー外で思考停止の既存テンプレを流用する

- 空状態のコピー：`"No items yet"` / `"プロファイルがありません"`
- エラー：`"Something went wrong"` / `"エラーが発生しました"`
- 汎用ストックイラスト・汎用ローディングスピナー
- リリースノート：`"バグ修正とパフォーマンス改善"`
- 初回起動：`"Welcome to Midori!"`

**AI の初稿をそのまま採用できる箇所こそ、人の手を入れる候補地。** 逆に、AI の初稿で十分な箇所（フォームラベル等）は無理に捻らない。労力配分の原則。

## 書き分け例

| 箇所 | 量産 UI | Midori |
|---|---|---|
| 空状態（プロファイル 0 個） | プロファイルがありません | まだ信号の通り道がありません。入力と出力をつないでみましょう |
| 接続エラー（非重大） | 接続エラー | VRChat が応答しません。OSC 送信先ポートが `9000` か確認してみてください |
| リリースノート | バグ修正とパフォーマンス改善 | `pulse` 型の出力が時々サボっていたので、ちゃんと仕事させました |
| 起動完了ログ | `bridge started` | `bridge up. listening on 2 inputs, 1 output.` |
| 重大エラー（書き分けない） | ワークスペースを読み込めません | ワークスペースを読み込めません（平易・具体優先を維持） |

---

## 運用

- 各画面の設計ドキュメントには、**装飾層・フロー外で遊ぶ箇所**を列挙する（該当がなければ「なし」と明記）。
- モックアップ時、機能契約層はメタファーから導出、装飾層は別のレビューポイントとして扱う。
- コピーレビューでは「AI の初稿をそのまま採用できるか？」を問い、採用できる箇所はフロー内なら OK、フロー外なら再考する。
